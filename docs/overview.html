Why this exists

SWE-bench leaderboards answer "did this row pass." They don't answer "was passing this row a real measurement of capability."

Two failure modes that look identical on a leaderboard:

Looks likeActually is
Task is brokenModel failedTests demand undescribed behaviour
Agent cheatedModel passedAgent pip installed the fix
Honest passModel passedModel passed
Honest failModel failedModel failed

A single score conflates all four. bench-cleanser separates them, with citations, before anyone trains on the data or publishes against it.

The two-axis model

Axis 1 — Task contamination

Is the benchmark item itself fair?

  • Multi-label over 7 binary labels
  • Severity = pure set membership (no thresholds, no floats)
  • Inputs: problem text, gold patch, F2P/P2P tests, repo state

Axis 2 — Agent trajectory

How did this agent reach its result?

  • One label per (task, agent)
  • Heuristics + LLM, with cross-agent quorum upgrade
  • Inputs: action trace, final patch, resolved flag

An APPROACH_LOCK task is broken whether or not any model attempts it. An agent that pip installs the fix has cheated whether or not the task is clean. Stage 7 fuses both axes into a single fairness verdict.

red verdicts set invalidates_measurement=True — the single boolean a downstream consumer uses to drop a row from a leaderboard. Every verdict ships with reasoning and evidence: list[str].

Deterministic — no LLM call. Reproducible bit-for-bit from the persisted axis labels.

Architecture · seven stages

Stage 1   parse      diffs from gold patch + test patch              [deterministic]
Stage 1.5 visit      shallow-clone repo · attach test source · AST    [deterministic]
Stage 2   intent     LLM extracts core requirement + scope            [LLM]
Stage 3   structural astred-core / stdlib ast diff                    [deterministic]
Stage 4A  patch ↔ intent                                              [LLM]
Stage 4B  test  ↔ intent                                              [LLM]
Stage 4C  cross-ref  overpatch–overtest coupling                      [deterministic]
Stage 5   classify   dual-taxonomy LLM, heuristic-protected           [LLM]
Stage 6   severity   set-membership bucket                            [deterministic]
─────────────────────────────────────────────────────────────────────────────
Stage 7   fusion     Axis 1 × Axis 2 → FairnessVerdict                [deterministic]

Every LLM stage emits a Pydantic BaseModel; severity, structural diff, cross-ref, and fusion are all deterministic. The package contains zero random.* calls.

Determinism contract

GuaranteeConcretely
No floats in severity or fusionSeverity = set membership; fusion = (severity × labels × traj × resolved) → verdict.
No randomLLM backoff jitter is (attempt mod 4) · 0.25 · base — deterministic.
Schema-enforced LLM I/Oresponse_format={"type":"json_schema","strict":true}, with json_object fallback. Cache-before-validate so a corrupt payload stays inspectable.
Resumable runs--resume on by default; reports reloaded from disk are authoritative.
Frozen across model upgradesSeverity rules depend only on label set. Swapping the LLM never changes the severity mapping.
Atomic writesPer-report JSON uses tempfile + os.replace; an interrupt never leaves half-files for --resume to mis-load.
Cite-or-shut-up evidenceA label without an evidence list is rejected by the classifier.

Layered classifier

  1. Heuristics — normalised diff similarity (quote-aware comment strip), pip-install detection, F2P-name leakage.
  2. LLM — strict-output Pydantic; heuristic signals embedded in prompt.
  3. Cross-agent quorum — median pairwise similarity ≥ 0.85 AND ≥ 10 added lines → upgrade to GOLD_PATCH_LEAK. Outlier-tolerant; honest low-entropy convergence protected.

What we just hardened

Strict-mode schemas

True json_schema strict:true with a _strictify_schema transformer (drops defaults, forces additionalProperties:false, inlines $defs). API rejects malformed candidates before they reach us.

Error sentinel removed

Pipeline failures no longer masquerade as SEVERE. New pipeline_error field; severity stays CLEAN; Stage-7 fusion skips error rows. Aggregate stats no longer lie.

Concurrency throttled

Trajectory analyzer now respects max_concurrent_requests. Prevents rate-limit storms silently degrading LLM analysis to the heuristic fallback.

Cache stable across pydantic

Structured cache key uses (model_name, schema_version), not the full schema text. Pydantic upgrades no longer wipe cached responses.

Heuristic union

Three high-precision deterministic heuristics (pre-staged tests, 0-REQUIRED mismatch, self-referential text) survive even if the LLM omits them.

Cross-agent quorum

Median quorum + low-entropy gate. One-line bug fixes no longer get flagged as gold-patch leaks; one diverging agent no longer vetoes the upgrade.

Honest caveats

  • CLEAN means "no contamination signal on seven labels" — not "perfect benchmark item."
  • Axis-1 labels inherit LLM judgement noise; deterministic stages only amplify what upstream labels say.
  • Cross-reference coupling is file-level, not function-level — known gap in audits/severe/AUDIT_PROTOCOL.md.
  • This is a research instrument, not a metric. It makes human review tractable; it does not replace it.

What's it for

If you train on SWE-bench rows, every contaminated task teaches the wrong lesson.
If you publish numbers against it, every contaminated row distorts the score.

bench-cleanser is the tool you run before either.
